# Public signal to canon pipeline

## Status

Accepted design posture.

## Purpose

This file defines how public user demand becomes Chummer-owned truth without letting public SaaS tools own product direction, support handling, release truth, or canon.

## Core rule

Public signal is input. Canon is decided by Chummer.

```text
ProductLift / Katteb / users
  -> EA synthesis
  -> Product Governor decision
  -> chummer6-design canonical patch
  -> implementation or generated public guide change
  -> chummer.run / ProductLift roadmap / ProductLift changelog closeout
```

## Decision classes

- support misroute
- docs/help gap
- feature idea
- horizon signal
- house-rule demand
- open-run demand
- BLACK LEDGER signal
- Community Hub signal
- mobile companion signal
- creator-publishing signal
- public content optimization
- rejected / out of scope
- duplicate / merged signal

## Public idea to shipped feature

```text
User posts/votes in ProductLift
  -> EA weekly digest
  -> Product Governor triage
  -> if simple: docs/code/design route
  -> if complex: Deftform/Icanpreneur/MetaSurvey/Lunacal discovery
  -> design patch or milestone
  -> implementation
  -> release or public availability proof
  -> ProductLift changelog closeout
  -> voters notified when allowed
```

## Public guide improvement

```text
Guide page underperforms or is confusing
  -> Katteb audit
  -> EA content packet
  -> design/hub review
  -> upstream canonical source edit
  -> public guide regenerated
  -> closeout if tied to public request
```

## KARMA FORGE demand

```text
ProductLift idea: "Let me change gear availability"
  -> Deftform pre-screen
  -> Icanpreneur interview
  -> HouseRuleDemandPacket
  -> EA cluster
  -> Product Governor classification
  -> KARMA FORGE candidate
  -> rule-environment/amend package design
  -> ProductLift status update
```

## BLACK LEDGER / Community Hub demand

```text
ProductLift idea: "Let GMs post public jobs"
  -> ProductLift votes/comments
  -> Deftform asks GM/player role and pain
  -> Icanpreneur interview if high-signal
  -> OpenRunDesignPacket
  -> design patch or implementation candidate
  -> if shipped, ProductLift changelog and voter notification
```

## Closeout requirements

For ProductLift-linked shipped work:

- public closeout packet exists
- release, guide, Hub route, or artifact proof exists
- public changelog entry exists or carries an explicit no-entry reason
- voter notification is allowed before ProductLift notification fires
- public roadmap, guide, support, and release surfaces do not contradict each other

For Katteb-linked content work:

- source packet exists
- accepted change lands upstream in design or a Chummer-owned source registry
- generated guide or public page is regenerated from Chummer-owned source
- reviewer and approval owner are recorded
- unsupported claims are rejected or removed

## Anti-patterns

Do not:

- let ProductLift become the real roadmap
- let votes override Product Governor review
- mark ProductLift items shipped before user availability
- turn support cases into ProductLift posts
- ask users to paste book text
- edit generated public guide output directly with Katteb
- auto-publish Katteb articles without Chummer review
- let Katteb invent unshipped features or rules claims
- use ProductLift comments as private support
- build a content calendar without owner and review cadence

## Success criteria

The pipeline is healthy when high-signal ideas become discovery packets, accepted findings become canon or explicit no-change decisions, users can follow public status without mistaking it for implementation priority, and every public “shipped” claim is backed by Chummer-owned evidence.
